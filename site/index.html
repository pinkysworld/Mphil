Explainable and Time-Aware Malware Family Classification

Studying how malware family classification behaves under temporal drift, leakage control, and feature fusion.

This project examines whether dynamic malware reports can support a more robust and interpretable family-classification pipeline when evaluation is aligned with time rather than relying only on random train/test splits.

View Repository See Workflow
Dataset Avast-CTU CAPEv2

Reduced dynamic-analysis reports with family labels, malware type, and detection dates.

Current Focus Baselines and temporal evaluation

Preparing the baseline pipeline before moving into deeper analytical comparisons.

Core Themes Leakage, drift, calibration

The project emphasises fair evaluation and interpretable model behaviour over time.

Research Questions

What is the project trying to answer?

RQ1

Which feature views retain predictive value when malware behaviour is evaluated across time rather than at random?

RQ2

How much do results change once family-name leakage and similar shortcuts are controlled?

RQ3

Does feature fusion remain useful under chronological evaluation and rolling-origin testing?

RQ4

Are the resulting class probabilities reliable enough for calibration and selective prediction analysis?

Method

How the workflow is structured

  1. Ingest and validate Unpack reports, align them to the label file, and confirm the SHA-256 join.
  2. Audit for leakage Check whether family names or related markers leak the answer into the input.
  3. Build evaluation splits Create random and chronology-aware splits for fair comparison.
  4. Extract feature views Derive API-token, artifact-token, behavioural-count, and static PE representations.
  5. Run baseline experiments Train linear or boosted baselines on single views or fused views and record the outputs.
  6. Analyse robustness Study temporal invariance, walk-forward behaviour, leakage ablation, and calibration.

Data and Reproducibility

What is public and what stays local

Public in the repository

The public repository contains the code, configuration, and project notes needed to reproduce the workflow structure.

  • Pipeline scripts and experiment runner
  • Working notes and project plan
  • GitHub Pages site and deployment workflow

Kept local

Raw archives, extracted reports, caches, and large result files are kept outside version control.

  • Avast-CTU reduced archive
  • Extracted JSON reports
  • Feature caches and processed data
  • Local experiment outputs

The reduced report archive is the current default because it keeps the same label definitions while remaining much easier to handle during baseline development and repeat runs.
