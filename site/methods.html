Study Design

Questions, workflow, features, and evaluation choices.

The study is structured around one main concern: how well malware family classification holds up once the evaluation setup starts to resemble the passage of time rather than a convenient random shuffle.

Questions Workflow Feature views Evaluation Working terms

Research Questions

Core questions for the project

RQ1

Which feature views remain informative when the evaluation is time-aware rather than random?

RQ2

How sensitive are the results to leakage controls and naming shortcuts in the reports?

RQ3

Does combining multiple evidence sources outperform single-view models under more realistic test settings?

RQ4

How reliable are the model probabilities once calibration and selective prediction are considered?

Workflow

Pipeline from raw reports to thesis figures

01

Ingest and validate

Match the report archive to the label file and confirm that the SHA-256 identifiers line up correctly.

02

Check for leakage

Inspect the reports for strings that would let a model recover the answer directly rather than learning behaviour.

03

Build the splits

Create random and chronological train, validation, and test partitions for fair comparison.

04

Extract features

Convert sandbox content into feature matrices covering API text, artifact text, behaviour counts, and PE metadata.

05

Train baseline models

Run the first classification experiments across single-view and fused feature settings.

06

Run robustness analyses

Measure temporal stability, drift, leakage sensitivity, and calibration after the baseline pass.

Feature Views

What the models are trained on

API TF-IDF

Resolved API names are treated as text and converted into sparse vectors. This captures calling patterns in the dynamic reports.

Artifact TF-IDF

File paths, mutexes, registry activity, and related strings provide another text-based view of the execution trace.

Behaviour counts

Simple numeric summaries such as counts of written files or observed APIs give a compact behavioural profile.

Static PE features

Section metadata, imports, exports, and other PE-derived fields provide a purely static complement to the dynamic views.

API hash

The hashed API representation offers a lighter-weight baseline when a full learned vocabulary is not required.

Fusion

Several feature families are concatenated so that the classifier can draw on multiple forms of evidence at once.

Evaluation

Why the project uses more than one test setup

Random stratified

Samples are shuffled while preserving class balance. This is useful as a reference point, but it often makes the task look easier than deployment would be.

Global chronological

Earlier samples are used for training and later samples for testing, which is closer to how a system would face future malware.

Per-family chronological

Each family is split internally by time so that the study can separate temporal change from simple class-availability effects.

Throughout the project, macro-F1 is treated as the main headline score because it gives each malware family equal weight rather than allowing larger classes to dominate the result.

Working Terms

Concepts used throughout the project

Dynamic report

A sandbox report records what a file did during execution, including API calls, file activity, and other observed behaviour.

TF-IDF

TF-IDF weights tokens by their importance to a sample relative to how common they are across the corpus.

SGD classifier

A fast linear model that works well with the large, sparse feature spaces produced by text-based representations.

Leakage

Leakage occurs when the input accidentally contains the answer, for example through family names or obvious detection labels.

Calibration

Calibration asks whether model confidence behaves as expected, rather than simply whether the top prediction is correct.

Selective prediction

Selective prediction studies whether a system can become safer by abstaining on low-confidence samples.
